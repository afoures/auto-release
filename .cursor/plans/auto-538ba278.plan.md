# Auto-release CLI Design & Implementation Plan

## 1. Define core domain model & public API

- Specify TypeScript interfaces in `src/index.ts` for:
- `AutoReleaseConfig`, `AppConfig`, `DeployConfig`, `VersionStrategy`, `ResolvedChange`.
- Implement and export public helpers:
- `defineConfig(config: AutoReleaseConfig): AutoReleaseConfig`.
- Built-in strategies: `semverStrategy`, `calverStrategy`.
- Core types (for user configs and custom strategies).
- Capture constraints:
- `apps[].packages`: list of folders each containing a `package.json`.
- Per-app `versioning` (strategy + allowed `changeTypes`).
- Root-level `changesDir`, `defaultChangelogDir`, and per-app `changelog.path`.
- Per-app `deploy` config supporting both command string and handler function.

## 2. Implement config loading (`auto-release.config.ts`)

- Default config file: `auto-release.config.ts` at repo root, with optional `--config <path>` override.
- Support only `export default` (no CommonJS), assuming Node >= 22.
- Implement a TS/ESM-aware loader that:
- Resolves the config path and dynamically imports it.
- Validates that the default export conforms to `AutoReleaseConfig` (runtime checks, helpful error messages).
- Normalize and freeze the loaded config for internal use.

## 3. Implement versioning strategies (semver, calver, custom)

- Define `VersionStrategy` interface:
- `id: string`, `changeTypes: readonly string[]`.
- `parse(version: string): unknown`, `format(parsed: unknown): string`.
- `bump({ currentVersion, changes, now }): string`.
- **Semver strategy**:
- Support change types `major | minor | patch | none`.
- Parse and format standard `x.y.z` strings.
- When multiple changes exist, bump by highest precedence (major > minor > patch > none).
- `none` contributes to changelog only and never bumps by itself.
- **Calver strategy**:
- Format `YYYY.MM.micro` with zero-padded month; `micro` is an integer starting at 0.
- On bump: if current `YYYY.MM` equals latest release’s `YYYY.MM`, increment `micro`; otherwise set `YYYY.MM` to current date and `micro` to 0.
- All pending changes (any type) cause a micro bump; types are for grouping only.
- **Custom strategies**:
- Expose `versionStrategies` registry in config where users can define custom strategies.
- Allow apps to reference either built-ins or custom strategies; validate at config load that strategies are well-formed.

## 4. Implement change file discovery and parsing

- Global structure:
- Root `changesDir` (default `.changes`).
- Per-app subfolder: `${changesDir}/${appName}`.
- Filenames:
- Enforce pattern: `type.slug-words.md`.
- Validate that `type` is in the app’s `versioning.changeTypes`.
- Markdown parsing rules:
- If first non-empty line starts with `#`, treat that heading as the title and remaining content as body.
- Otherwise, treat the first line as a simple title with no body.
- Represent each change as a `ResolvedChange` containing:
- `appName`, `type`, `title`, optional `body`, and `filePath`.
- Implement helpers to list pending changes for a given app and to compute whether an app has any pending changes.

## 5. Implement package & version resolution per app

- For each `AppConfig`:
- Resolve each `packages[]` folder to a `package.json` file.
- Read and parse all `package.json` files.
- Verify that all packages for an app share the same `version` value; otherwise surface a clear validation error.
- Implement helpers to:
- Get `currentVersion` per app.
- Write a new version string to all related `package.json` files for that app.

## 6. Implement changelog writer

- Determine each app’s changelog path:
- If `app.changelog.path` is set, use it.
- Otherwise, use `${defaultChangelogDir}/${app.name}.md`.
- Implement a function that, given `{ app, currentVersion, nextVersion, date, changes[] }`:
- Formats a new release section header like `## 1.3.0 – 2025-11-26` or `## 2025.11.0 – 2025-11-26`.
- Groups changes by `type` with headings (e.g. `### Major`, `### Minor`, `### Patch`, `### None` for semver; `### Feature`, `### None` for calver).
- Renders each change as a bullet with its title and optional body.
- Inserts/appends to the changelog file in a stable, idempotent way.

## 7. Implement CLI framework and shared utilities

- Introduce a CLI entry module (e.g. `src/cli.ts`) and expose it via `bin` in `package.json` as `auto-release`.
- Use a minimal CLI framework (e.g. `commander`) or a simple custom parser to define subcommands:
- `validate`, `change`, `preview`, `release`, `deploy`.
- Implement shared utilities:
- Global flag parsing: `--config`, `--dry-run`, `--yes`, `--json` (where relevant).
- Consistent logging and error reporting helpers.
- Common routine to load config, resolve apps, and collect change files.

## 8. Implement `validate` command

- Command: `auto-release validate [--config <path>] [--json]`.
- Steps:
- Load config via the loader and validate schema and invariants (apps, packages, strategies).
- For each app, verify all `packages[]` exists and that their `package.json` versions match.
- Scan `changesDir` and per-app subfolders; validate filenames, types, and that markdown can be parsed.
- Output:
- Human-readable summary: per-app status and list of problems.
- With `--json`, output a structured report: `{ valid, errors: [...], warnings: [...] }`.
- Exit with non-zero status if any validation errors occur (CI-friendly).

## 9. Implement `change` (create change) command

- Command: `auto-release change [--app <name>] [--type <changeType>] [--summary <text>] [--description <text>]`.
- Behavior:
- If `--app` not provided, prompt user to pick from configured apps.
- If `--type` not provided, prompt from that app’s allowed `changeTypes`.
- If `--summary` not provided, prompt a single-line summary.
- If `--description` not provided, optionally prompt for multi-line description (ended by blank line or EOF).
- Create change file:
- Ensure directory `${changesDir}/${appName}` exists.
- Generate a slug (e.g. `YYYY-MM-DD-add-user-profile-xyz`).
- Filename: `<type>.<slug>.md`.
- Content:
- If only summary: single-line markdown.
- If description: `# <summary>` plus paragraphs for `description`.
- Print the created path to stdout for scripting.

## 10. Implement `preview` command

- Command: `auto-release preview [--app <name>]`.
- Behavior:
- Load config and current versions for all apps.
- For each app (or filtered by `--app`), gather pending changes from `.changes/<appName>/`.
- If an app has pending changes, compute `nextVersion` via its `VersionStrategy.bump`.
- Output (human readable only for v1):
- For each impacted app: list app name, current version, next version, and each pending change with `[type] title (filePath)`.
- No filesystem writes; this is purely informational and used as a dry-run preview of the next release state.

## 11. Implement `release` command

- Command: `auto-release release [--app <name>] [--dry-run] [--yes]`.
- Behavior:
- Load config, apps, current versions, and pending changes.
- Determine target apps (all, or one via `--app`).
- For each target app with pending changes:
- Compute `nextVersion` via its strategy.
- If `--dry-run`:
- Print a detailed plan of what would change (per app: current → next version, files to update, changelog path, change files to consume).
- If not `--dry-run`:
- Unless `--yes` or non-interactive, show a summary and ask for confirmation once.
- Update `version` field in every `package.json` under `app.packages`.
- Append a new section to the app’s changelog using the changelog writer.
- Handle change files (e.g. delete or move to an `_applied` archive folder, depending on chosen initial behavior; document this default in README).
- Exit non-zero if any write fails.
- No git commits or tags; this command is strictly version/changelog + change-file management.

## 12. Implement `deploy` command with git tagging

- Command: `auto-release deploy [--app <name>] [--dry-run] [--yes]`.
- Assumptions:
- Usually run on CI after `release` has updated versions and changes are pushed.
- Behavior:
- Load config and resolve target apps.
- For each target app:
- Read current version from its `package.json` files (already aligned by `release`).
- Build a deployment context `{ app, currentVersion, packages, dryRun, logger, exec }`.
- If `deploy.command` is set and not `--dry-run`, run the shell command.
- If `deploy.handler` is set, call it with the context.
- Track per-app deployment success or failure.
- If **any** app deployment fails:
- Do not create git tags.
- Exit with non-zero status.
- If **all** targeted apps succeed and not `--dry-run`:
- Create git tags using `git.tagTemplate` for each app (e.g. `${appName}@${version}`).
- Optionally create a git commit or leave commits entirely to the user/CI (documented behavior; default is tags only).
- With `--dry-run`:
- Print which commands/handlers and tags would be executed, but do not run them.

## 13. Testing and examples

- Add unit tests for:
- Version strategies (semver and calver bump logic).
- Change file parsing and filename validation.
- Config loading and per-app package/version resolution.
- Changelog generation formatting.
- Add basic CLI tests (or integration-style tests) for `validate`, `change`, `preview`, and `release` flows.
- Provide example `auto-release.config.ts` configs for:
- Single-app repo (one package).
- Monorepo app composed of multiple packages.
- Mixed semver and calver apps with custom strategy.

## 14. Documentation

- Update `README.md` to include:
- Overview of the tool and philosophy (Changesets-inspired, app-centric).
- Installation and Node version requirement (>= 22).
- Config file format (`auto-release.config.ts`) with full examples.
- Explanation of `.changes/<appName>/` layout and filename scheme.
- Detailed command reference with examples for all CLI commands.
- Recommended CI workflow tying together `validate`, `preview`, `release`, and `deploy`.

## 15. Future enhancements (not in initial scope)

- Optional JSON output for `preview` and `release` impacts.
- More advanced dependency graph handling and internal version range updates.
- App grouping helpers beyond multiple `packages[]` entries.
- Pluggable changelog templates and localization support.
